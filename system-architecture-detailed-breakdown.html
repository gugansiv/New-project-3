02
Application & Business Logic Layer
The backend services that enforce rules, process data, and connect the portals to the database

This layer is built as a set of focused services rather than one giant codebase. Each service owns one job — orders, payroll, stock, etc. — so a bug or slowdown in, say, the notification service can never bring down order-taking. Communication between services and the front-end happens over a REST API; some live features (order queue updates) use WebSockets for instant push instead of the front-end having to repeatedly ask "anything new?".

Node.js + NestJS REST API (JSON over HTTPS) WebSocket (Socket.io) — live order queue & notifications Redis — caching & session store JWT — auth tokens Docker containers, one per service

Why NestJS over plain Express: it enforces a consistent structure (modules, controllers, services) across a system this size, which matters once more than one developer is touching the codebase — plain Express gets messy fast at this scope without that structure.

03
Data Layer — The Database(s)
Where everything actually lives, and how it stays safe and audit-proof

Not one database doing everything — two physically separate database instances. This split exists specifically so the audit log can be made tamper-proof: it sits where even a Super Admin's application credentials have no delete/update rights, enforced at the database permission level, not just by app-level rules that a bug or insider could bypass.

Primary Operational Database
PostgreSQL 16, Multi-AZ
  • All live business data: users, orders, stock, finance, payroll, pricing, loyalty
  • Strong relational consistency — critical for financial accuracy
  • Row-level security policies enforce branch/region data isolation at the DB level, not just app code
  • Automatic synchronous replica in a second data center (this is what gives zero-downtime failover)
Audit Log Database
PostgreSQL, separate instance, append-only
  • Single table design: who, what, when, before-value, after-value
  • Application's DB user has INSERT and SELECT only — no UPDATE, no DELETE, enforced by Postgres GRANT permissions
  • Physically separate instance so a breach of the main DB doesn't expose write access here too
  • Indexed by user, branch, action type, timestamp for fast filtering in the Admin audit viewer
Cache & Session Layer
Redis
  • Active login sessions / JWT blacklist (for instant logout/account suspension)
  • Cached product prices & menu (fast reads, refreshed on price change)
  • Real-time order queue state for WebSocket push
File & Object Storage
S3-compatible object storage
  • Menu item photos, receipt/expense photo uploads, issue report attachments
  • Payslip PDFs, generated financial reports

Database Permission Reality Check
Two database users, two different rights — enforced by Postgres itself.
The application connects to the primary DB with a user that has normal read/write. It connects to the audit DB with a different database user that Postgres has only granted INSERT and SELECT to — no GRANT for UPDATE or DELETE exists at all. This means even if there were a bug in the application code that tried to edit a past audit record, the database itself would refuse the query before it ever happened. That's what makes "non-editable" a real guarantee instead of just a policy.
How to use this document: This breakdown is structured so each portal's feature list and each service's endpoint list can be lifted directly into sprint tickets. Recommended build order: Auth + Audit Log Service first (everything else depends on them) → Inventory + Order Service → Finance/Payroll Service → Loyalty Service → Notification Service last (least urgent, easiest to bolt on). Tech choices above (Node.js/NestJS, PostgreSQL, Redis, React) reflect standard, well-supported 2026 tooling for this exact category of system — real-time order flow plus strong financial consistency — rather than experimental or niche frameworks, so hiring developers and finding documentation will be straightforward.
